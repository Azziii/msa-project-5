# Задание 2. Разработка дизайна модуля пакетной обработки данных

## Исходные данные

Каждое утро в 06:00 необходимо формировать персонализированные CSV/XLS-прайс-листы для B2B-клиентов на основе данных из PostgreSQL.

Таблицы:

* products (5 000–10 000 строк)
* categories (50–200 строк)
* clients (100–500 строк)
* client_prices (10 000–20 000 строк)

Обработка данных сводится к выполнению SQL JOIN и выгрузке результата в файл. Сложная бизнес-логика отсутствует.

## Выбранное решение

Для реализации выбран Kubernetes CronJob.

### Обоснование

Причины выбора:

* задача запускается один раз в сутки;
* отсутствуют сложные ETL-процессы;
* объём данных относительно небольшой;
* инфраструктура уже работает в Kubernetes;
* отсутствует необходимость в DAG-оркестрации;
* минимальные требования к вычислительным ресурсам;
* простое сопровождение и эксплуатация.

Использование Apache Airflow, Spring Batch или Spark приведёт к усложнению архитектуры без существенных преимуществ.

## Принцип работы решения

1. Kubernetes CronJob запускается ежедневно в 06:00.
2. Контейнер генератора прайс-листов подключается к PostgreSQL.
3. Выполняется SQL JOIN таблиц:

   * products
   * categories
   * clients
   * client_prices
4. Для каждого клиента формируется персональный CSV/XLS.
5. Файлы сохраняются в объектное хранилище (S3/MinIO).
6. Логи собираются в ELK.
7. Метрики экспортируются в Prometheus.
8. После завершения Job контейнер удаляется.

## Вывод

Наиболее рациональным решением является Kubernetes CronJob как наиболее простой, дешёвый и естественный способ реализации периодической пакетной выгрузки данных в Kubernetes-инфраструктуре.
