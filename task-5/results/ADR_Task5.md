# Название задачи

Внедрение централизованного мониторинга, логирования и оповещения для Spring Batch ETL-платформы TradeWare

## Контекст

Компания TradeWare планирует увеличение нагрузки на систему обработки складских отчётов.

Текущая инфраструктура не предоставляет:

* централизованного мониторинга;
* централизованного логирования;
* системы оповещений;
* возможности оперативного поиска причин ошибок.

Для решения задачи требуется внедрить мониторинг приложения и инфраструктуры, а также централизованный сбор логов.

## Принятое решение

Для мониторинга выбраны:

* Prometheus;
* Grafana.

Для логирования выбраны:

* Filebeat;
* Logstash;
* Elasticsearch;
* Kibana.

Источником метрик является Spring Boot Actuator с экспортом метрик в формате Prometheus.

Источником логов является Logback.

## Обоснование выбора мониторинга

Prometheus:

* является стандартом мониторинга облачных приложений;
* поддерживает Pull-модель;
* хорошо интегрируется со Spring Boot;
* поддерживает Alerting Rules.

Grafana:

* предоставляет визуализацию метрик;
* поддерживает оповещения;
* имеет готовые дашборды.

## Выбранные метрики

### JVM

* jvm_memory_used_bytes
* jvm_memory_max_bytes
* jvm_threads_live_threads

### Процесс

* process_cpu_usage
* system_cpu_usage

### HTTP

* http_server_requests_seconds

### Spring Batch

* успешные выполнения Job
* неуспешные выполнения Job
* длительность выполнения Step

## Обоснование логирования

В качестве решения выбран ELK Stack.

Преимущества:

* централизованное хранение логов;
* полнотекстовый поиск;
* поддержка фильтрации;
* удобный анализ ошибок Batch Job;
* готовность к микросервисной архитектуре.

Для доставки логов используется Filebeat.

## Оповещения

Настроены два оповещения:

1. Ошибка выполнения Batch Job.
2. Использование памяти JVM более 80%.

## Риски

* Elasticsearch требует значительных ресурсов.
* При росте количества логов потребуется настройка ротации индексов.
* При увеличении числа сервисов потребуется стандартизация логирования.
