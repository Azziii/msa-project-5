# Решение

## Анализ проблемы

Текущая система выполняет обработку файлов синхронно внутри монолитного приложения.

Это приводит к:

* блокировке пользовательских операций;
* высокой нагрузке на PostgreSQL;
* отсутствию пакетной обработки;
* отсутствию механизма восстановления после ошибок;
* невозможности масштабирования ETL независимо от ERP.

## Принятое решение

В качестве ETL-фреймворка используется Spring Batch.

Архитектура обработки:

1. Пользователь загружает CSV.
2. ERP сохраняет файл в GCS.
3. ERP инициирует Batch Job.
4. Spring Batch выполняет:

   * Reader;
   * Processor;
   * Writer.
5. Processor получает дополнительные данные из таблицы loyality_data.
6. Writer пакетно сохраняет записи в таблицу products.
7. Результат доступен аналитическим сервисам.

## Причины выбора Spring Batch

* Java уже используется компанией.
* Поддерживает Chunk Processing.
* Поддерживает Retry и Restart.
* Подходит для ETL-задач.
* Хорошо работает с PostgreSQL.
* Легко контейнеризуется в Docker.
* Может быть перенесён в Kubernetes/GCP.

# Альтернативы

## Apache Airflow

Преимущества:

* развитая оркестрация процессов.

Недостатки:

* требует отдельного ETL-компонента.

## Apache Spark

Преимущества:

* высокая производительность.

Недостатки:

* избыточен для объёма до 1 млн строк в сутки.

## Kafka Streams

Преимущества:

* event-driven архитектура.

Недостатки:

* усложняет систему;
* хуже подходит для пакетной обработки CSV.

# Недостатки выбранного решения

* ориентирован на batch, а не streaming;
* требуется отдельное управление Job Execution;
* при росте нагрузки свыше десятков миллионов записей может потребоваться Spark.
